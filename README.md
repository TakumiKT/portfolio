## サービス概要

登録販売者、登録販売者資格取得に向けて勉強している人の為の、

向けの業務メモ作成アプリです。

## 開発背景 ##

ドラッグストアで副店長として働く中、多くのOTC医薬品(一般用医薬品)の相談をお客様から受けました。

中には、登録販売者資格の学習でも会社研修でも学ばなかった相談内容も多く有りました。

上長に聞いたり、自分で調べる等していく中で、これを手軽にPC上で纏めて後から振り返る事が出来たらなと感じていました。

その経験から登録販売者向けに医薬品情報・接客事例・注意点を簡単に記録できるアプリを作成したいと考えに至りました。



## ターゲットとするユーザー層について ## 

本アプリは、登録販売者及びその学習者を対象にしています。

主にドラッグストア従業員や、転職・復職に向けて接客判断を整理したい方にもおすすめです。

本アプリは特定店舗への導入を前提とせず、登録販売者個人での利用を想定しています。


## サービスの利用イメージ ## 

登録販売者が日々の業務の中での疑問点やナレッジを記録、振り返りをするときに利用する事をイメージしています。

- 接客時に判断が難しかったケースを、業務後に振り返りたいとき
- 同じような症状への対応を、過去の記録から確認したいとき
- 自分の接客判断の傾向や弱点を整理したいとき


## サービスの差別化ポイント・推しポイント ## 

* Microsoft Teams
* Notion

これら2つのアプリは幅広い用途に対応できる一方で、

特定の情報を振り返ろうとした際に、見つける事に手間取る等の課題があります。

本アプリの特徴は、登録販売者が接客時に行っている判断プロセスを、

あらかじめ決められた観点で分解して記録できる点にあります。


一般的なノートアプリでは、

「接客内容を自由に書く」ことはできても、

どの観点で振り返ればよいかは利用者に委ねられています。

本アプリでは、接客メモを以下のような構造化された項目で記録します。


- 相談された症状・状況

- 接客時に確認したポイント

- 自分が行った判断・対応内容

- 注意した点や迷った点

- 振り返って感じた課題や改善点


これらにより、単なる記録ではなく、

「なぜその判断をしたのか」「どこで迷ったのか」を

後から振り返りやすい形で知識として蓄積することができます。

登録販売者が自身の接客判断を言語化し、経験を学習につなげることに特化している点が、

既存の汎用ノートアプリとの大きな違いです。


### MEMOTCの使い方

MEMOTCは、登録販売者の接客メモを「思考プロセスに沿って」構造化して残し、あとから検索・振り返りしやすくする個人向けアプリです。


### MEMOTCでできること

- 接客メモを「相談内容 → 確認したこと → 判断 → 注意点 → 振り返り」で記録
- キーワード検索（症状・判断内容・タグ など）
- タグで分類・絞り込み（例：かぜ / 小児 / 禁忌確認）
- お気に入りで「重要メモ」を集約
- 下書き→清書の運用（接客直後は下書き、後で整理して清書）
- 期間指定・並び順などの高度検索
- 生成AI（OpenAI API）による、作成したメモ内容に対するフィードバックとレポート作成(各1日3回まで)



### 基本の使い方（4ステップ）

1. ログイン
- ログイン後、メモ一覧が表示されます。
2. 新規メモを作成
- 「新規メモ」から、接客内容を入力します。
3. タグを付けて分類
- タグ欄にカンマ区切りで入力します（例：かぜ, 小児, 禁忌確認）。 あとから一覧でタグクリック → 絞り込みできます。
4. 一覧で振り返り・検索
- キーワード・タグ・期間・お気に入り・下書き/清書を組み合わせて、過去の判断をすぐ見つけられます。


### おすすめのタグ設計

- 症状：頭痛 / 咳 / 鼻水 / のど など
- 対象：小児 / 高齢 / 妊娠授乳 など
- 注意：禁忌確認 / 併用注意 / 受診勧奨 など
- 目的：経過観察 / セルフケア / 相談メモ など


### 入力例（構造化のイメージ）

「どう書けばいいか分からない」場合は、まずこの形を真似してみてください。

- 相談内容・症状：喉の痛み。発熱はなし。2日前から。

- 接客時に確認したポイント：妊娠授乳なし／持病なし／服薬なし。のどの腫れ・呼吸苦なし。

- 自分の判断・対応内容：まずはうがい・保湿。悪化時は受診を案内。市販薬は成分を確認して提案。

### 一覧画面での検索・絞り込み（ボタンの意味）

一覧画面では「検索 → クリア → 下書き/清書 → お気に入り」の順に操作すると迷いません。

- 検索：キーワード（症状・判断内容・タグ名など）で絞り込みます
- クリア：絞り込み条件をリセットして一覧に戻します
- 下書き：下書きのメモだけを表示します（接客直後のメモ整理に便利）
- 清書：清書（公開）のメモだけを表示します（通常は清書がデフォルト表示）
- お気に入り：重要メモだけを表示します


### 高度検索（より絞り込みたいとき）

- 検索対象： 全体 / 症状 / 判断 から選べます。
例：判断だけを検索したい場合は「判断」を選んでキーワードを入力します。
- 開始日 / 終了日：期間で絞り込めます。
例：今月の接客メモだけ見たい場合は、開始日と終了日を設定します。
- 並び順： 新しい順 / 古い順 を選べます。
キーワード検索・タグ絞り込み・下書き/清書・お気に入りは、併用できます。

条件が付いているときは、画面上部の条件チップの「×」から個別に解除できます。


### テンプレート作成

- 作成/編集/削除： 新規メモ作成画面の上部から作成できます。
あらかじめテンプレートを作成することで、似た内容のメモを簡単に作成することができます。

## 採用理由（Why this stack）

### Rails / PostgreSQL / Puma
- **Rails**：CRUD・認証・Active Storage・I18n など、必要機能が標準で揃っており、短期間で安定した実装ができるため。
- **PostgreSQL**：本番環境（Render）でも一般的で、検索・JOIN・制約（ユニーク制約等）を堅牢に扱えるため。
- **Puma**：Railsの定番アプリサーバで、RenderなどのPaaSでも運用実績が多いため。

### アセット / フロント（Propshaft / Importmap / Hotwire）
- **Propshaft**：Rails 8 の標準的なアセット管理として採用し、構成をシンプルに保つため。
- **Importmap**：NodeやWebpack等のビルド環境を最小化し、学習コストと運用コストを下げるため。
- **Hotwire（Turbo/Stimulus）**：SPAのような操作感を、Railsのサーバサイド中心で実現できるため（フォーム送信・画面遷移・軽いUI制御に強い）。

### JSON（Jbuilder）
- 将来的な拡張（API化・外部連携）を見据えて、Rails標準のJSON生成手段として採用。

### 起動高速化（Bootsnap）
- 開発・CIでの起動時間を短縮し、生産性を上げるため。

### 画像アップロード（Active Storage / image_processing）
- **Active Storage**：Rails標準で実装でき、ユーザーアイコン等の添付管理が容易なため。
- **image_processing**：画像のリサイズ（variant）等を安定して行うため。

### 認証 / OAuth（Devise / Google OAuth）
- **Devise**：実装が枯れており、登録・ログイン・編集などを安全に短時間で整備できるため。
- **Google OAuth（omniauth-google-oauth2）**：入力の手間を減らし、ユーザーが利用開始しやすくなるため。
- **omniauth-rails_csrf_protection**：OAuthフローでのCSRF対策を適切に行うため。

### 日本語化（rails-i18n / devise-i18n）
- UI文言を日本語で統一し、想定ユーザーにとって迷いの少ない体験にするため。
- 文字列を一元管理して、保守性（変更のしやすさ）を上げるため。

### AI（openai）
- **OpenAI API**：接客メモの振り返り支援（フィードバック/レポート生成）を実現し、アプリの差別化ポイントを強化するため。
- 併せて、**回数制限・入力最小化・結果キャッシュ**などを入れ、コスト/安全面にも配慮。

### Rails 8 DB-backed adapters（solid_cache / solid_queue / solid_cable）
- **DBベース**で完結できるため、Redis等の追加インフラを避けつつ、ジョブ/キャッシュ/リアルタイム機能の土台を作れる。
- 小規模ポートフォリオとして、構成の複雑化を抑えながら拡張性を確保できる。

### HTML解析（Nokogiri）
- テストや補助処理（HTMLからトークン取得など）で必要になった際に、定番ライブラリとして採用。

### 開発 / テスト / 品質（RSpec / RuboCop / Security tools）
- **RSpec / FactoryBot / Faker**：モデル・リクエスト・サービスのテストを書きやすくし、品質を担保するため。
- **Capybara / Selenium**：ユーザー操作（画面遷移・フォーム送信）のE2E確認を自動化するため。
- **RuboCop（+ omakase）**：可読性とスタイル統一を行い、レビュー・保守性を高めるため。
- **Brakeman / bundler-audit**：脆弱性検出をCIに組み込み、最低限のセキュリティ対策を継続的に行うため。
- **debug / web-console**：開発中の原因切り分けを効率化するため。
